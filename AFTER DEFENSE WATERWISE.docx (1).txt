INTRODUCTION
12
The deployment of gamified educational applications in developing regions is
critically hindered by the hardware and network constraints of the target environment.
A primary technical challenge is the prohibitive energy cost of standard Dynamic
Difficulty Adjustment (DDA) systems, which frequently utilize deep learning models
ill-suited for battery-dependent devices. Jung et al. (2025) report that the local
processing required for these complex algorithms can increase energy consumption
by up to 94%. This high energy consumption directly exacerbates the issue of
thermal throttling on legacy Android architectures. Benoit-Cattin et al. (2020)
demonstrated that sustained CPU load from heavy inference tasks triggers protective
thermal mechanisms within seconds, forcing the operating system to drastically lower
clock speeds to prevent hardware damage.
This thermal generation poses a direct threat to system usability. Kang et al.
(2019) found that high-load processing raises device temperatures to uncomfortable
levels, with 75% of users reporting disrupted interaction. When thermal limits are
breached, the resulting throttling can increase "frame rate defects" or lag spikes by
up to 146% (Sahin et al., 2015). In an interactive context, this latency results in
"unfair player failure," where a user fails not due to a lack of skill, but because the

2
device could not render the frame in time. Consequently, there is a critical need for
lightweight, heuristic-based alternatives specifically engineered to maintain O(1)
constant-time complexity to minimize this energy overhead and ensure stable game
loops on edge devices.
Furthermore, the reliance on unstable mobile ad-hoc networks (MANETs) for
multiplayer interaction introduces severe data consistency challenges. Guidec et al.
(2022) established that standard synchronization protocols and JSON-based
Conflict-Free Replicated Data Types (CRDTs) suffer from metadata bloat, which
renders them highly vulnerable to the frequent packet loss characteristic of
opportunistic networks. This creates a specific need for an optimized integer-only
synchronization mechanism capable of ensuring state convergence without the
bandwidth overhead inherent in existing libraries.

Objectives of the Study
The general objective of this study is to develop "WaterWise," a rule-based adaptive
difficulty game with offline multiplayer capabilities, to evaluate the computational
efficiency of deterministic heuristics in edge computing environments.
Specifically, the study aims to
1.​ design and implement a deterministic Rule-Based Rolling-Window Adaptive
Difficulty Algorithm to replace computationally expensive deep learning
models, thereby mitigating thermal throttling and energy drain on legacy edge
devices.
2.​ develop an Integer-Only G-Counter CRDT synchronization mechanism to
resolve metadata bloat and ensure state convergence across high-loss,
opportunistic Mobile Ad-Hoc Networks (MANETs), and
3.​ evaluate the computational performance of the optimized algorithms by
measuring frame rate stability, memory allocation consistency, and network
payload efficiency under simulated hardware constraints.

Scope and Limitations
The proposed study focuses on the design, development, and algorithmic
validation of WaterWise, a 2D cross-platform educational game specifically
engineered for users in resource-constrained edge computing environments. The
system is built using the Godot 4.5 Game Engine to ensure deployment viability on
resource-constrained hardware, specifically targeting legacy Android devices
(Cortex-A53 architecture, <2GB RAM) and standard Windows workstations found in
Philippine public school laboratories.
Presentation Module. The Presentation Module encompasses the graphical
user interface (UI) and user experience (UX) components, including the Initial
Screen, Main Menu, In-Game HUD, and Character Customization interfaces. This
module is designed to process touch inputs for mobile devices and mouse/keyboard
inputs for desktop environments. The visual scope is strictly limited to 2D
sprite-based animations and lightweight UI elements to strictly enforce the 16ms
frame budget required for 60 FPS on low-end hardware, it will not support 3D

7
rendering, dynamic lighting, or particle-heavy effects that trigger thermal throttling.
Furthermore, while the interface supports real-time English and Filipino localization
via a hash-map lookup system, it does not include support for other regional dialects
or assistive technologies for visually impaired users beyond basic text scaling.
Core Game Logic Module. The Core Game Logic Module defines the core
gameplay

mechanics

through

a

modular

inheritance

structure, utilizing a

MiniGameBase abstract class to govern the logic of specific mini-games such as
CatchTheRain, BucketBrigade, FixLeak, and SortWater. This module handles
lightweight

physics

collisions

(AABB/Circle),

score

calculation,

and

timer

management using deterministic logic. The scope is confined to finite, arcade-style
gameplay loops designed for short burst sessions (1-3 minutes), it excludes
open-world exploration, complex rigid-body physics simulations, or infinite procedural
generation to maintain constant-time O(1) computational complexity.
Service Module. The Service Module functions as the system’s central
intelligence, implementing the Rule-Based Rolling-Window Adaptive Difficulty
Algorithm and the G-Counter Conflict-Free Replicated Data Type (CRDT) for network
synchronization. The adaptive system calculates a Proficiency Index (Φ) based on a
fixed-size circular buffer of the last five games, utilizing weighted moving averages
and standard deviation for consistency penalties. A critical limitation is that this
module explicitly excludes Deep Learning, Neural Networks, or AI-based inference to
minimize battery drain and thermal throttling. Additionally, the NetworkManager is
restricted to serverless, peer-to-peer UDP communication (Port 7777) over local
ad-hoc networks (MANETs), it does not support wide-area network (WAN)
connectivity, centralized cloud servers, or anti-cheat enforcement mechanisms.
Data Persistence Module. The Data Persistence Module manages the local
storage and retrieval of application data, utilizing the Godot ConfigFile API for user
preferences (volume, language) and generating JSON exports for gameplay

8
telemetry and session performance metrics. The system operates on a Local-First
Architecture, ensuring that all progress and logs are saved directly to the device’s
internal storage. Consequently, the study limits data handling to offline contexts, it will
not support cloud database integration (e.g., Firebase, AWS), cross-device save
synchronization, or remote analytics dashboards. Security is limited to local file
permissions and binary serialization, without advanced encryption standards for the
exported JSON logs.

Definition of Terms
The following terms are defined as used within the context of this research
Adaptive Difficulty Algorithm. A computational system that automatically adjusts
game challenge levels based on real-time analysis of player performance data,
utilizing rolling-window methodology to assess recent results.
Conflict-Free Replicated Data Type (CRDT). A distributed data structure enabling
strong eventual consistency across multiple independent nodes without centralized
coordination,

allowing

peer-to-peer

data

synchronization

through

mathematically-proven commutative operations.
Flow State. The optimal engagement zone where challenge perfectly matches skill
level, representing the sweet spot between anxiety and boredom where individuals
realize peak performance and deep engagement.
G-Counter Algorithm. A grow-only counter CRDT implementation enabling
conflict-free score synchronization across distributed multiplayer devices with O(1)
time complexity for increment operations.

9
GDScript. Python-like programming language designed for ease of use in the Godot
game engine, facilitating rapid development while maintaining code clarity and
maintainability.
Godot 4.5. An open-source, cross-platform game development engine specifically
selected for its flexibility, scene-based modular architecture, and suitability for
educational game development.
Greywater. Wastewater generated from household activities such as laundry,
dishwashing, and bathing that can be safely reused for non-potable purposes like
toilet flushing and plant watering.
ISO/IEC 25010 Software Product Quality Model. An international standard for
evaluating software quality across eight characteristics, functional suitability,
performance efficiency, compatibility, usability, reliability, security, maintainability, and
portability.
Mini-game. A short, focused game activity that presents a specific water
conservation challenge within the larger educational game, designed to take 1-3
minutes to complete.
Proficiency Index (Φ). A unified metric combining accuracy and consistency
performance that serves as the primary decision variable for adaptive difficulty
adjustments, ranging from -0.2 to 1.0.
Rolling Window. A data analysis technique that maintains only the five most recent
performance records while discarding older data that no longer reflects current ability,
ISOoperating through FIFO (first-in, first-out) mechanisms.
Rule-Based System. A computational approach that applies predetermined logical
conditions (if-then rules) to make decisions, offering transparency and explainability
compared to complex machine learning models.

10
Water Reuse. The practice of using water more than once for the same or different
purposes, such as collecting rainwater for cleaning or using vegetable washing water
for plants.
Zone of Proximal Development (ZPD). The space where tasks are beyond current
abilities but attainable with the right level of challenge, where the brain must stretch,
adapt, and develop new strategies without hitting the frustration wall.

REVIEW OF RELATED LITERATURE

​

This chapter presents the key discussion, concepts, and sources that will

form both the information and theoretical foundations of this study. It will include the
supporting literature and related works that help strengthen the direction of the
research. The reviewed materials in this chapter serve as guides in shaping the
research methods, as well as in outlining the objectives and overall flow of the study.

Energy-Efficient Algorithms vs. Deep Learning on Mobile
Green computing minimizes energy consumption which is critical for mobile
gaming where high CPU usage depletes batteries. While Dynamic Difficulty
Adjustment maintains engagement existing solutions rely on expensive AI models
that impose heavy loads unsuitable for low-end devices (Marmat & Thankachan
2025). Kim et al. (2021) found that AI-driven thermal management introduces its own
energy overhead while Kang et al. (2019) notes that algorithmic complexity causes
thermal spikes. This study addresses that gap by replacing deep learning with a
deterministic Rule-Based Rolling Window Algorithm effectively eliminating the "AI tax"
on battery life.
The Impact of Game Loops on Thermal Throttling
Mobile games rely on a continuous Game Loop that keeps processors active.
Rua and Saraiva (2023) found that high frame rates are the primary battery drain
while complex logic causes thermal runaway exceeding 45°C. Kang et al. (2019).
notes this triggers throttling and stuttering. Consequently Energy-Aware loops using
O(1) logic are essential to allow the CPU to sleep between frames.

12
Thermal Constraints and Algorithmic Efficiency on Edge Devices
Recent literature emphasizes the link between algorithmic complexity and
mobile thermal limits. Benoit-Cattin et al. (2020) revealed that Deep Learning triggers
thermal throttling on uncooled devices causing immediate performance degradation.
Sahin et al. (2015) showed that this throttling increases lag spikes by 146%
disrupting the user experience. Furthermore Kang et al. (2019) found that high-load
tasks raise device temperatures over 50°C causing physical discomfort for 75% of
users.
Mobile Ad-Hoc Networks (MANETs) and Synchronization
Mobile Ad-Hoc Networks in P2P gaming face challenges like variable latency
and packet loss. Dantas and Baquero (2025) showed that CRDTs reduce latency by
75% compared to servers but suffer from metadata bloat that clogs bandwidth.
Saquib et al. (2022) argued that standard CRDTs prioritize theoretical consistency
over practical throughput in unstable networks. Sypytkowski (2021) notes that even
optimized versions carry heavy overhead for causality tracking. This validates the
need for Integer-Only Payload Optimization which strips metadata to ensure
convergence even during the severe packet loss scenarios common in opportunistic
network settings.
Algorithmic Efficiency and Memory Management
Implementing a First-In, First-Out (FIFO) queue via a fixed-size circular buffer
is highly optimal for edge computing because it strictly enforces a maximum size limit
without ever requiring dynamic memory resizing. By maintaining a fixed memory
footprint, this data structure guarantees constant-time O(1) complexity for insertion
and deletion operations. According to Algocademy (2025), circular buffers are ideal
for real-time systems as they provide constant-time operations and ensure memory
safety.

13
Furthermore, Upadhya et al. (2025) noted that sliding window mechanisms
outperform traditional stream mining in terms of memory efficiency. Crucially, this
architectural decision prevents heap fragmentation and eliminates the need for the
garbage collector to execute, the primary mechanisms that trigger latency spikes and
thermal accumulation on devices with constrained RAM.
Peer-to-Peer Architecture for Latency Reduction
For multiplayer functionality in areas with poor internet connectivity a
Peer-to-Peer or P2P architecture offers distinct advantages over traditional
client-server models. Yahyavi and Kemme (2025) in their survey of Massively
Multiplayer Online Games found that P2P architectures effectively reduce latency by
removing the server hop and allowing clients to communicate directly. However they
noted that state consistency remains a challenge in these decentralized systems.
Recent research confirms that Conflict-Free Replicated Data Types or CRDTs allow
for eventual consistency without the heavy bandwidth overhead of locking
mechanisms which makes them ideal for the high-packet-loss environments typical of
mobile hotspots (Guidec et al., 2022).
Addressing Android Ecosystem Fragmentation
The diversity of the Android ecosystem poses a unique challenge known as
fragmentation where an application must run on thousands of different device
configurations. Trio Content Team (2025) reports that as of late 2025 fragmentation
remains a critical issue with only 4.5% of devices running the latest Android version
while a vast majority rely on older and less secure iterations. This diversity forces
developers to optimize for the lowest common denominator to avoid crashes and
performance inconsistencies. Alphabin (2025) further highlights that fragmentation
increases the risk of device-specific defects where an app works on one phone but
fails on another due to differing screen sizes or chipsets.

14
Algorithmic Foundations of Rolling Window Heuristics
​

Selecting the right data structure is critical for maintaining O(1) time

complexity. Algocademy (2025) confirms that Circular Buffers provide constant-time
operations ideal for real-time systems. Mees (2018) validates that Weighted Rolling
Averages estimate learner ability more accurately than raw scores. While Lopes et al.
(2025) note a lack of lightweight frameworks this research bridges that gap using
Fixed-Size Circular Buffers. This approach prevents garbage collection spikes and
frame drops on Android devices.
Portability and Cross-Architecture Optimization
The Android ecosystem suffers from extreme hardware fragmentation
between ARM and x86 architectures causing significant performance variance
(Intelivita, 2025). Godot 4's Vulkan renderer struggles on low-end devices despite
abstraction. Therefore efficient mobile development requires optimizing underlying
logic using lightweight algorithms like Rolling Window and Integer-Only CRDT to
reduce workload. This ensures playability on diverse low-cost hardware found in
Philippine schools by minimizing computational overhead. (Quinnox, 2025)
The Application Domain: Water Conservation and Gamification
While the primary focus of this study is algorithmic optimization, the
application domain of water conservation was selected due to urgent environmental
necessities. The Philippines faces intensifying water stress projected to worsen from
climate change, creating a critical need for effective educational intervention.
Shohieb et al. (2025) validated that adaptive serious games significantly improve
learning outcomes by keeping learners in their "Zone of Proximal Development," a
finding supported by Chiotaki et al. (2023), who noted that gamification enhances
retention compared to traditional methods.

15
To ensure this pedagogical validity applies locally, the game content was
derived from established environmental curricula and expert consultations. Baseline
domain research revealed critical misconceptions, notably that a significant portion of
the target demographic could not distinguish safe from unsafe water sources,
necessitating specific mini-games like SortWater for classification logic. Furthermore,
regional preferences indicated a need for Filipino learning materials. Consequently,
this study implements the adaptive benefits identified by Shohieb et al. within a
bilingual framework, ensuring the technical artifact is both culturally responsive and
pedagogically grounded.
Educational Game-Based Learning Focused on Values
The "Ugaling Pinoy" mobile learning game demonstrates that culturally
contextualized educational games effectively foster values education and behavioral
change among Filipino elementary students through interactive gameplay featuring
quizzes, fill-in-the-blank challenges, and scenario-based activities that facilitate both
knowledge acquisition and positive attitudinal shifts (Escultor, Moises, & Sales,
2023).
Collaborative Environmental Education through Games
Environmental education games promote ecological behaviors by combining
hands-on learning with immersive environments (Tsai et al. 2025). Integrating
community activities with games fosters lasting habits (Abrencia 2025) while
collaborative gameplay strengthens environmental reasoning and cognitive outcomes
(Janakiraman et al. 2021).
Rolling Window Rule-Based Adaptive Difficulty Algorithms.
The rolling window technique keeps learners in their Zone of Proximal
Development outperforming static difficulty (Shohieb et al. 2025). Rule-based
systems offer transparency and stability valuable for education. However Lopes et al.
(2025) note a gap in applying this to 2D games on resource-constrained devices

16
while reviews suggest using collective metrics to maintain motivation (Chiotaki et al.
2023).
Theoretical Foundations of Conflict-Free Replicated Data Types (CRDTs).
Conflict-free Replicated Data Types enable seamless replication without
coordination bottlenecks ensuring deterministic convergence through strong eventual
consistency (Preguiça et al. 2018). State-based approaches using delta transmission
are particularly useful for Philippine schools as they reduce bandwidth overhead
while maintaining consistency (Shapiro et al. 2011 Almeida et al. 2018). This
robustness is proven in major systems like League of Legends.
G-Counter Algorithm and Distributed Counting Semantics.
The G Counter is a state based CRDT using a local array where replicas
increment only their own entry. Merging takes the element wise maximum and sums
entries for a global count guaranteeing convergence. This offers low latency and
partition tolerance for resource constrained educational systems without heavy
infrastructure (Preguiça et al 2018). Limitations include increment only support and
array growth with replicas making it best for stable groups like school sessions while
PN Counters better suit dynamic membership or decrements (Preguiça et al 2018).
Recent Advances in CRDT Implementation for Real-Time Distributed Systems.
Contemporary research focuses on optimizing CRDTs for practical use.
Dantas and Baquero (2025) demonstrated that CRDTs support responsive real time
VR by treating latency as a data consistency issue rather than just a network
problem. Their work on operation based CRDTs using vector clocks and state based
CRDTs with delta transmission offers valuable insights for multiplayer educational
games. Additionally industrial systems like Yjs often outperform academic prototypes
by utilizing specialized data structures and realistic testing methods (Preguiça et al.,
2018).

17
State-Based and Operation-Based CRDT Implementation in Educational
Gaming.
State-based CRDTs reconcile differences by comparing complete state
vectors ensuring results remain independent of message order but requiring higher
bandwidth (Sypytkowski 2021). Operation-based CRDTs transmit only incremental
operations to reduce storage but shift complexity to the replication protocol to ensure
causal ordering (Shapiro et al. 2011).
Implementation Challenges and Practical Considerations.
Dantas and Baquero (2025) documented challenges in distributed systems
and network fairness within BrickSync yet performance analysis showed latency
averaging 50 milliseconds a 75% reduction compared to remote servers. G-Counter
CRDTs track player proficiency by counting successes without synchronizing
complex values (Shapiro et al. 2011). However ensuring consistent calculations
requires carefully designed algorithms particularly when one player's level affects
shared parameters (Dantas & Baquero 2025).
Peer-to-Peer Architectures for Multiplayer Gaming.
Peer-to-peer architectures distribute tasks to improve scalability but require
latency management (Jiang 2012). Research shows client-server models suit
competitive games while P2P fits cooperative educational apps. Modern hybrid
systems balance server load and individual tracking by combining server authority
with P2P synchronization (Viitanen 2025).
Distributed State Synchronization Mechanisms for Collaborative Gameplay.
Multiplayer games require consistent state maintenance across devices using
models like server authoritative or lockstep depending on the genre (Kroupp 2024).
However cooperative educational games can tolerate temporary divergence making
CRDT-based synchronization an ideal solution. Because CRDT updates converge
deterministically without centralized servers they effectively address the infrastructure
constraints found in Philippine schools (Dantas & Baquero 2025).

18
Multiplayer Collaborative Learning and Educational Game Design.
Collaborative learning research proves peer interaction drives deeper
understanding than isolated study. Multiplayer games leverage this by improving self
management through shared objectives while MMORPG systems support real time
collaboration (Velásquez et al. 2024 Ye et al. 2025). Furthermore integrating adaptive
difficulty tailors challenges to individual skills enabling students of varying abilities to
collaborate effectively.
Multiplayer Collaborative Gaming and Educational Benefits.
Multiplayer educational games enhance learning by fostering social
collaboration and critical thinking. Studies confirm that peer interaction drives better
attitudinal outcomes than traditional instruction (Hatta et al. 2024 Janakiraman et al.
2021). Furthermore integrating cooperative elements with meaningful progression
systems significantly increases student motivation and completion rates (Canella et
al. 2025).

Synthesis
The review of literature confirms two distinct technical problems in edge
computing, specifically the energy inefficiency of deep learning models and the
bandwidth inefficiency of standard synchronization protocols in MANETs. A clear gap
exists regarding the replacement of AI with deterministic heuristics to explicitly solve
thermal throttling in offline educational games. Furthermore, there is a lack of
simplified, integer-only CRDT implementations designed specifically for high-loss
opportunistic networks. This research bridges these gaps by engineering the
WaterWise system, which replaces AI inference with an O(1) Rule-Based Adaptive
Engine to address energy deficits and replaces JSON CRDTs with a Binary
G-Counter to resolve bandwidth constraints.

METHODOLOGY

This research employs a Scrum-Agile development methodology to design,
construct and rigorously validate the WaterWise system utilizing a seven-sprint
iterative lifecycle to isolate and solve specific computational challenges found in edge
computing environments. The methodology is fundamentally experimental, focusing
on the construction of two novel algorithmic artifacts, a Rule-Based Rolling-Window
Adaptive Difficulty engine and a G-Counter Conflict-Free Replicated Data Type
(CRDT) for synchronization. Unlike traditional waterfall approaches, the Agile
framework allows for the continuous profiling of algorithmic complexity against the
hardware constraints of low-end Android devices enabling the research team to
rigorously enforce an O(1) time complexity budget throughout the development
process. By adopting this approach the study systematically isolates technical
variables specifically thermal throttling and network instability to demonstrate that
deterministic heuristics can outperform deep learning models in terms of energy
efficiency and frame rate stability addressing the critical performance gaps identified
by Rua and Saraiva (2023) regarding mobile application energy consumption.
Research Design
Target Testing Environment
The technical validation of the system targets the computational constraints of
the edge deployment environment rather than qualitative educational outcomes. For
system testing, human participants are utilized strictly as a source of stochastic input
data. Their interaction with the system generates the high-variance telemetry
specifically erratic reaction times and frequent input errors required to stress-test the
adaptive difficulty algorithm's ability to filter noise and detect proficiency shifts under
real-world conditions. By treating these users as stochastic data generators, the
research can rigorously evaluate the algorithm's ability to filter outlier data and detect

22
genuine algorithmic proficiency shifts without the computational overhead of complex
neural networks.
Research Design and Approach
The study utilizes an Experimental Software Engineering design, explicitly
centered on algorithmic performance analysis and system optimization. The primary
mode of inquiry is Constructive Research, where the core contribution is the technical
artifact itself specifically, the optimized algorithms embedded within the application.
This design operationalizes the software system not merely as an interactive tool, but
as a technical testbed to rigorously evaluate two specific engineering hypotheses
under simulated hardware stress the thermal stability of rule-based heuristics and the
network resilience of integer-only CRDT payloads.

Rule-Based Rolling-Window Adaptive Difficulty Algorithm
Rolling Window Data Structure. The algorithm utilizes a First-In-First-Out
(FIFO) queue data structure of size five, maintaining a rolling window containing the
five most recent mini-game performance records while discarding older records that
no longer reflect current student capability. This specific window size of five was
selected through careful balance of two competing design considerations reflecting
fundamental tradeoffs in adaptive system design.
Smaller windows with size less than three enable rapid response to
performance changes but create erratic difficulty shifts where students quickly
advance and regress repeatedly in oscillating patterns, creating frustration and
cognitive instability as students rapidly face changing challenge levels. Larger
windows with size greater than seven provide stable performance assessment but
result in sluggish adaptation that frustrates learners whose capabilities have
genuinely improved, maintaining overly simplified tasks despite demonstrated
mastery.
Contemporary research on dynamic difficulty adjustment demonstrates the
effectiveness of rolling window approaches in maintaining optimal engagement, with
Lin et al. (2025) finding that rolling window mechanisms with window sizes between
3-7 represent optimal balances between data stability and responsiveness. The size
of five represents this empirically-validated balance, providing sufficient data for
reliable performance assessment through averaging across multiple games while
maintaining responsiveness to genuine capability changes occurring as students
learn through gameplay. The rolling window data structure operates through FIFO
mechanisms where new performance records are added to the back of the queue
and oldest records are automatically removed from the front when the window

30
exceeds size five. Conceptually, the window slides forward with each new game
completion, always retaining the five most recent games' data while discarding
games from more than five games ago. This design ensures that the assessment
basis evolves continuously as students continue gameplay, reflecting current
capability rather than being weighted toward historical performance from early in the
session when students had less experience. The rolling window approach
fundamentally differs from cumulative averaging approaches that weight all historical
data equally regardless of recency, recognizing that recent performance provides
more valid indicators of current capability than historical performance from the
beginning of the session.

Adaptive Difficulty Mechanism and Flow Theory. The adaptive difficulty
system is grounded in established psychological principles of optimal engagement
and learning. Shohieb et al. (2025) describe how dynamic difficulty adjustment
(DDA) automatically adjusts scenarios, parameters, and game behaviors in real time
based on the player's skill level. By comparing a player's current performance to an
ideal or reference performance (such as the average first-time performance on a
level), the game dynamically modifies difficulty settings. The system maintains
optimal balance between challenge and engagement, avoiding boredom when tasks
are too easy or frustration when tasks are too difficult.

G-Counter CRDT Multiplayer Synchronization Algorithm
The G-Counter (Grow-only Counter) CRDT sub-module governs the offline
multiplayer synchronization process, ensuring conflict-free score convergence
without centralized server coordination. The operational process of this sub-module
initiates immediately when a local player successfully completes a game event. The
sub-module executes a local increment operation in constant time, updating only the
player's specific assigned index within the local state array. To transmit this updated
state across the Mobile Ad-Hoc Network (MANET), the sub-module executes a
data-packing process that explicitly strips all string-based metadata. It packs the
updated integer array into a highly optimized binary payload, which is then broadcast
via the NetworkManager over UDP Port 7777 to the connected peer device.
Upon receiving an incoming network payload from a peer, the local G-Counter
sub-module initiates the synchronization process by executing a deterministic merge
operation. Rather than simply overwriting local data or relying on timestamp-based
conflict resolution, the sub-module processes the data through an element-wise
mathematical comparison between the incoming remote array and the existing local
array. By executing a MAX() function to select the highest known integer for each
specific index, the process automatically resolves state conflicts. This mathematically
guarantees that even if transmission packets arrive completely out of sequence or
are temporarily lost due to severe network instability, any subsequent successful
payload delivery will force both devices to instantly converge to an identical, globally
synchronized score state.

directly correlates to higher transmission success rates and lower latency jitter in
environments characterized by high signal interference and up to 90% packet loss.

System Architecture
The WaterWise system architecture relies on a highly interrelated flow of data
across four specific layers, operating through a continuous feedback loop. The
process begins in the Presentation Layer, which is directly responsible for rendering
the visual interface and capturing user interactions. Within this layer, the UI Screens
sub-module handles system navigation and character customization, passing user
preferences downward. During gameplay, the In-Game HUD sub-module actively

46
captures touch or mouse events, instantly forwarding these raw inputs to the lower
layers while simultaneously receiving score and timer updates back from the system
to update the visual display. The Multiplayer UI sub-module manages lobby creation
and passes direct peer-to-peer connection requests to the network services.
Raw user inputs flow directly into the Core Game Logic Layer, which strictly
processes the spatial and mathematical mechanics of the educational simulations.
The MiniGameBase.gd sub-module acts as an abstract foundation, providing the
standard lifecycle processes (initialization, active loop, termination) for all activities.
The specific Mini Games sub-modules (such as CatchTheRain or BucketBrigade)
inherit this base class to execute unique scenario rules. The Game Logic sub-module
continuously computes 2D collision detection and triggers raw score tallies based on
valid player interactions.

47

Figure 9. WaterWise Four-Module System Architecture
Once a game loop concludes, the performance telemetry flows into the
Service Layer, functioning as the system's central intelligence and state coordinator.
The GameManager.gd sub-module acts as the primary router, capturing the raw
accuracy and time data and routing it to the AdaptiveDifficulty.gd sub-module. This
adaptive engine processes the data through the deterministic rolling-window
computation to immediately output calibrated difficulty parameters (speed multipliers,
time limits) for the next round. Concurrently, the NetworkManager sub-module utilizes
an ENet UDP protocol on port 7777 to broadcast local score events, while the
G-Counter CRDT sub-module actively processes incoming network payloads,
mathematically merging local and remote arrays to ensure global score convergence
without a centralized server. To ensure cultural responsiveness, the Localization.gd

48
sub-module operates globally across these processes, intercepting text rendering
requests and returning the correct English or Filipino strings via a hash-map lookup.

Finally, the processed data flows into the Data Persistence Layer for
permanent, offline storage. The ConfigFile sub-module processes the current
difficulty states and user preferences, serializing them into a local .cfg file to ensure
progress is retained between sessions. Simultaneously, the JSON Export
sub-module aggregates the session's performance telemetry and writes it securely to
the device's internal storage, completing the architectural loop and allowing
researchers to extract the raw metrics for analysis.

Performance Evaluation
To explicitly address the technical deliverables outlined in the study's
objectives, the evaluation of the system's computational performance relies on
specific profiling methods and mathematical computations conducted during
simulated 30-minute stress tests on legacy edge devices. Frame rate stability is
evaluated using the Godot Engine Profiler and the Android Studio native profiler. The
computation focuses on measuring the exact render time per frame (Tframe). For a
stable 60 Frames Per Second (FPS) experience, the system must compute and
render all logic within a strict 16.6-millisecond budget, ensuring Tframe <= 16.6 ms. The
method involves continuously logging frame times and calculating the percentage of
frames that successfully meet this budget versus dropped frames, mathematically
proving that the O(1) heuristic logic prevents the CPU load spikes that trigger
hardware thermal throttling. To validate the space complexity of the Rule-Based
algorithm, memory allocation consistency is monitored using the Godot Memory
Profiler. The computation measures the active Random Access Memory (RAM)
usage in Megabytes (MB) over the duration of the stress test. By calculating the
variance in memory allocation (ΔRAM) post-initialization, a successful evaluation

55
requires demonstrating that the fixed-size circular buffer maintains a flat, constant
memory footprint where ΔRAM = 0. This proves the system successfully avoids
dynamic memory reallocations and the subsequent garbage collection spikes that
cause system latency. Finally, network payload efficiency is evaluated to validate the
lightweight nature of the synchronization mechanism using network sniffing tools
such as Wireshark. The computation involves measuring the exact size, in bytes, of
the custom Integer-Only G-Counter CRDT binary payload over UDP. This value is
mathematically compared against the byte size of a standard JSON-serialized
payload containing identical score data. The method calculates the exact bandwidth
reduction percentage using the formula (SizeJSON - SizeBinary) / SizeJSON) x 100. This
computation rigorously validates the system's viability in opportunistic Mobile Ad-Hoc
Networks (MANETs) characterized by severe packet loss.
Instrumental Profiling and Thermal Stress Testing Protocols
To rigorously quantify the green computing efficiency claims, the study moves
beyond gameplay metrics to strictly hardware-level profiling. The evaluation
operationalizes thermal throttling not as a subjective feeling of lag, but as a
measurable reduction in CPU clock speed triggered by heat accumulation. The
device's thermal state is monitored using the Android Profiler and Godot Monitor to
log the CPU Temperature (T_cpu) at 1-second intervals. The primary evaluation
metric for throttling is defined as Clock Speed Stability (S_clk), where a drop in CPU
frequency below the manufacturer's optimal rated speed constitutes a throttling
event. The Rule-Based Algorithm is considered successful if it avoids triggering
protective underclocking over a continuous 30-minute stress test session,
demonstrating sustained thermal equilibrium compared to traditional architectures.
Simultaneously, energy consumption metrics are captured by measuring
battery drain in milliampere-hours (mAh) normalized per minute of gameplay. This

56
average discharge rate (AE) is compared against a baseline simulation running a
standard TensorFlow Lite MobileNet model to represent the Deep Learning
alternative. The system must demonstrate a measurable reduction in discharge rate
to empirically validate the efficiency findings of Jung et al. (2025) regarding heuristic
optimizations.
Computational Cost Analysis - Heuristics vs. Deep Learning
This section defines the algorithmic process that ensures frame rate stability
by mathematically comparing the instruction count required for decision-making. A
standard neural network used for dynamic difficulty adjustment typically requires
matrix multiplications for input processing. Even a small network requires
approximately O(N2) floating-point operations (FLOPs) per inference. On low-end
ARM Cortex-A53 processors, these continuous FLOPs generate significant heat,
leading to the thermal throttling identified by Kang et al. (2019).
In contrast, the proposed Rule-Based Rolling Window algorithm utilizes purely
arithmetic operations and conditional logic. The Sliding Window update operates in
O(1) time because it involves exactly one array insertion and one deletion, regardless
of session length. By replacing quadratic matrix math with constant-time pointer
arithmetic, the CPU remains in a low-power state for the majority of the frame
duration, waking only to process the lightweight physics engine. This specifically
addresses the Frame Rate Stability objective, ensuring the main thread never
exceeds the 16.6ms budget required for 60 FPS.
Network Fault Injection Protocol
To evaluate the G-Counter CRDT against the network instability variable, the
study employs a fault injection methodology using standard network conditioner
tools. The system is subjected to a controlled stochastic network profile mimicking
opportunistic edge environments by introducing randomized UDP packet drops and
latency spikes (jitter). The primary metric for this evaluation is Convergence Time

57
(T_conv), defined as the time elapsed between a client action and the Host state
reflecting that action. The G-Counter CRDT is evaluated on its ability to rapidly
achieve state convergence without relying on the re-transmission overhead that
typically congests standard JSON/TCP synchronization in high-loss environments,
thereby proving its viability for offline-first multiplayer edge gaming.

Complexity is evaluated through Big-O analysis of critical operations (time and space
complexity), with the specific success criterion that the rolling window update, CRDT
merge, and Proficiency Index (Φ) calculations all achieve O(1) complexity.

targeting less than 1 KB per sync packet and a maximum of 1 sync per second.

Ethical Considerations
